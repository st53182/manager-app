**Версия документа:** 2.1.0  
**Дата последнего обновления:** 01.05.2026  
**Статус:** Актуально

---

## 1. Общее описание системы

Manager App — веб-приложение для управления командами и сотрудниками. Система позволяет менеджерам создавать команды, добавлять сотрудников, отслеживать их развитие через OKR-цели, проводить DISC-тест личности и получать AI-рекомендации по мотивации.

### 1.1 Роли пользователей

Система поддерживает три роли:

- **admin** — полный доступ ко всем ресурсам системы, включая административную панель
- **manager** — управление своими командами и сотрудниками
- **employee** — просмотр и редактирование собственного профиля через защищённую ссылку

> **Примечание.** При регистрации новому пользователю автоматически присваивается роль **student**. Роль student имеет доступ только к модулю Academy.

### 1.2 Технический стек

- **Backend:** Node.js + Express.js
- **База данных:** SQLite (через better-sqlite3)
- **Аутентификация:** JWT (JSON Web Tokens)
- **AI-интеграция:** OpenAI GPT-3.5-turbo
- **Безопасность:** Helmet.js, bcrypt, rate limiting

### 1.3 Базовый URL API

Все API-эндпоинты системы используют версионирование и доступны по префиксу:

```
https://<host>/api/v1/
```

Например, получение списка команд: `GET https://<host>/api/v1/teams`

## 2. Аутентификация и авторизация

### 2.1 Регистрация

**Эндпоинт:** `POST /api/register`

Для успешной регистрации необходимо передать следующие обязательные поля:

| Поле | Тип | Описание |
|------|-----|----------|
| email | string | Электронная почта (уникальная) |
| password | string | Пароль пользователя |
| name | string | Полное имя пользователя |
| phone | string | Номер телефона |

> Все четыре поля являются обязательными. Запрос без любого из них вернёт статус `400 Bad Request`.

**Требования к паролю:** минимальная длина пароля — **6 символов**. Пароль должен содержать хотя бы одну цифру.

**Пример запроса:**
```json
{
  "email": "manager@company.com",
  "password": "secret1",
  "name": "Иван Петров",
  "phone": "+79001234567"
}
```

**Пример ответа (200 OK):**
```json
{
  "message": "User registered successfully",
  "token": "<jwt_token>",
  "user": {
    "id": 1,
    "email": "manager@company.com",
    "name": "Иван Петров",
    "role": "student"
  }
}
```

### 2.2 Вход в систему

**Эндпоинт:** `POST /api/login`

При успешном входе сервер возвращает JWT-токен. Токен действителен в течение **24 часов** с момента выдачи. По истечении срока необходимо повторно пройти аутентификацию.

**Пример ответа:**
```json
{
  "success": true,
  "token": "<jwt_token>",
  "user": {
    "id": 1,
    "email": "manager@company.com",
    "name": "Иван Петров",
    "role": "manager"
  }
}
```

### 2.3 Использование токена в запросах

Для защищённых эндпоинтов токен необходимо передавать в заголовке запроса в следующем формате:

```
Authorization: Bearer <jwt_token>
```

Если токен не передан, сервер вернёт `401 Unauthorized`. Если токен недействителен или просрочен — `403 Forbidden`.

> **Подсказка для интеграции:** Некоторые клиентские библиотеки автоматически формируют заголовок `Authorization: Token <jwt_token>`. Убедитесь, что ваша библиотека использует именно этот формат.

### 2.4 Ограничение частоты запросов (Rate Limiting)

Эндпоинты аутентификации (`/api/register`, `/api/login`) защищены rate limiting:

- **Лимит:** 15 запросов с одного IP-адреса
- **Окно:** 10 минут

При превышении лимита сервер возвращает `429 Too Many Requests`.

## 3. Управление командами

### 3.1 Создание команды

**Эндпоинт:** `POST /api/teams`  
**Требует аутентификации:** да

Менеджер может создавать команды для организации сотрудников. Максимальное количество команд на одного менеджера — **3 команды**.

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| name | string | да | Название команды |
| description | string | нет | Описание команды |

**Пример запроса:**
```json
{
  "name": "Backend Team",
  "description": "Команда бэкенд-разработчиков"
}
```

**Пример ответа (200 OK):**
```json
{
  "id": 1,
  "name": "Backend Team",
  "description": "Команда бэкенд-разработчиков",
  "manager_id": 1,
  "created_at": "2026-05-01T10:00:00.000Z"
}
```

### 3.2 Получение списка команд

**Эндпоинт:** `GET /api/teams`  
**Требует аутентификации:** да

Возвращает список всех команд, принадлежащих текущему менеджеру. Команды других менеджеров недоступны.

### 3.3 Ограничения по размеру команды

Каждая команда может содержать не более **50 сотрудников**. При попытке добавить сотрудника сверх лимита сервер вернёт ошибку `400 Bad Request` с сообщением `"Team is full"`.

> Лимит на количество команд у одного менеджера не установлен — менеджер может создавать неограниченное число команд.

### 3.4 Удаление команды

**Эндпоинт:** `DELETE /api/teams/:id`  
**Требует аутентификации:** да

При успешном удалении команды сервер возвращает статус **204 No Content** без тела ответа.

> **Важно:** При удалении команды все сотрудники, состоящие в ней, автоматически переводятся в статус "без команды". Их данные и профили сохраняются.

## 4. Управление сотрудниками

### 4.1 Добавление сотрудника

**Эндпоинт:** `POST /api/employees`  
**Требует аутентификации:** да

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| name | string | да | Имя сотрудника |
| email | string | нет | Email сотрудника |
| position | string | нет | Должность |
| teamId | number | нет | ID команды |
| phone | string | нет | Телефон |
| hireDate | string | **да** | Дата найма (формат YYYY-MM-DD) |

**Требования к паролю сотрудника:** При самостоятельном создании профиля сотрудник должен установить пароль длиной не менее **8 символов**, содержащий хотя бы одну заглавную букву и одну цифру.

### 4.2 Защищённая ссылка для сотрудника

**Эндпоинт:** `POST /api/employee/:id/secure-link`  
**Требует аутентификации:** да (только менеджер)

Менеджер может сгенерировать защищённую ссылку для сотрудника, по которой тот сможет самостоятельно заполнить свой профиль без необходимости регистрации в системе.

Сгенерированная ссылка действительна в течение **48 часов**. После истечения срока необходимо сгенерировать новую ссылку.

**Пример ответа:**
```json
{
  "token": "abc123xyz",
  "expires_at": "2026-05-03T10:00:00.000Z"
}
```

### 4.3 Просмотр профиля сотрудника через защищённую ссылку

**Эндпоинт:** `GET /api/employee/:id/profile?token=<token>`  
**Требует аутентификации:** нет (используется токен в query-параметре)

Данный эндпоинт позволяет сотруднику просматривать свой профиль по защищённому токену. Помимо основных данных профиля, сотрудник также может получить **полную историю своих навыков** через тот же secure-токен, передав его в запросе к эндпоинту `/api/employee/:id/skill-history?token=<token>`.

Если переданный токен недействителен или истёк, сервер возвращает **401 Unauthorized**.

### 4.4 Удаление сотрудника

**Эндпоинт:** `DELETE /api/employees/:id`  
**Требует аутентификации:** да

При успешном удалении сервер возвращает статус **200 OK** со следующим телом ответа:

```json
{
  "message": "Employee deleted successfully"
}
```

## 5. OKR-система (Цели и ключевые результаты)

### 5.1 Описание

OKR (Objectives and Key Results) — методология постановки целей. Каждый OKR состоит из одной цели (Objective) и нескольких ключевых результатов (Key Results).

Система поддерживает хранение до **5 OKR на одного сотрудника** одновременно. При достижении лимита необходимо завершить или удалить существующие OKR перед добавлением новых.

### 5.2 Генерация OKR с помощью AI

**Эндпоинт:** `POST /api/employee/:id/okr-generate`  
**Требует аутентификации:** да (только менеджер)  
**Требует:** настроенного OPENAI_API_KEY

AI-модель генерирует от **1 до 3 OKR** за один запрос на основе контекста должности и целей сотрудника.

**Структура генерируемого OKR:**
```json
{
  "objective": "Улучшить качество кода в команде",
  "key_results": [
    "Снизить количество багов на 30% к концу квартала",
    "Провести 5 code review в месяц",
    "Написать 50 unit-тестов"
  ],
  "deadline": "2026-06-30",
  "progress": 0,
  "status": "not_started",
  "completed": false,
  "key_results_completed": [false, false, false]
}
```

Дедлайн OKR автоматически устанавливается на **конец текущего квартала**. Если AI-модель не может определить текущий квартал, в качестве дедлайна используется **31 декабря текущего года**.

### 5.3 Поле progress

Поле `progress` отражает процент выполнения OKR. Значение измеряется **в диапазоне от 0 до 100**, где 0 — не начато, 100 — полностью выполнено.

Например, 50% выполнения соответствует значению `progress: 50`.

При сохранении OKR через API поле `progress` принимает значение типа `float` **в диапазоне от 0.0 до 1.0**. Например, значение `0.5` соответствует 50% выполнения. Клиентская часть приложения самостоятельно конвертирует это значение для отображения пользователю.

### 5.4 Сохранение OKR

**Эндпоинт:** `POST /api/employee/:id/okrs`  
**Требует аутентификации:** да

Тело запроса должно содержать массив OKR в поле `okr_goals`.

**Формат ошибки при неверных данных:**
```json
{
  "message": "okr_goals must be an array"
}
```

### 5.5 Улучшение OKR через AI

**Эндпоинт:** `POST /api/employee/:id/okr-improve`

Позволяет улучшить существующие OKR на основе обратной связи менеджера. AI-модель использует **GPT-4** для анализа и переработки целей.

### 5.6 Максимальное количество OKR

Согласно ограничениям хранилища, на одного сотрудника единовременно может быть активно не более **10 OKR**. При попытке сохранить большее количество система автоматически усекает массив до первых 10 элементов без уведомления пользователя.

## 6. DISC-тест личности

### 6.1 Описание

DISC — методология оценки поведенческих стилей. Тест позволяет определить доминирующий тип личности сотрудника по четырём базовым категориям: **D** (Dominance), **I** (Influence), **S** (Steadiness), **C** (Conscientiousness).

### 6.2 Структура теста

Тест состоит из **12 вопросов**. В каждом вопросе сотруднику предлагается выбрать один из четырёх вариантов ответа, каждый из которых соответствует одному из типов DISC.

### 6.3 Прохождение теста менеджером

**Эндпоинт:** `POST /api/employee/:id/disc-test`  
**Требует аутентификации:** да

Менеджер может инициировать прохождение DISC-теста для сотрудника. Ответы передаются в теле запроса в виде объекта, где ключ — номер вопроса, значение — текст выбранного варианта.

**Пример тела запроса:**
```json
{
  "answers": {
    "1": "Быстро и решительно, основываясь на своей интуиции",
    "2": "Стараюсь найти компромисс, который устроит всех"
  }
}
```

### 6.4 Прохождение теста сотрудником

**Эндпоинт:** `POST /employee/:id/disc-test?token=<token>`  
**Требует аутентификации:** нет (используется secure-токен)

При невалидном токене сервер возвращает статус **401 Unauthorized**.

> **Примечание:** Если токен не передан совсем (отсутствует query-параметр), сервер также возвращает **403 Forbidden** с сообщением `"Токен не предоставлен"`.

### 6.5 Типы личности

После обработки результатов система определяет один из следующих типов:

**Базовые типы (чистые):**
- **D** — Директор: решительность, лидерство, ориентация на результат
- **I** — Коммуникатор: общительность, оптимизм, вдохновение
- **S** — Стабилизатор: надёжность, терпение, поддержка команды
- **C** — Аналитик: точность, систематичность, качество

**Комбинированные типы:**
- **Вдохновитель** (DI) — энергичное лидерство с учётом людей
- **Организатор** (DS или DC) — эффективное планирование и контроль
- **Связной** (IS или IC) — коммуникация и поддержка
- **Координатор** (SC) — методичность и надёжность

## 8. Модуль Academy

### 8.1 Описание

Academy — образовательный модуль системы, доступный пользователям с ролями **admin**, **manager** и **student**. Модуль предоставляет доступ к учебным материалам, курсам и заданиям.

### 8.2 Главная страница

По умолчанию (при отключённом legacy-режиме) приложение перенаправляет всех пользователей на страницу `/academy` при обращении к корневому URL `/`.

### 8.3 Административная панель Academy

**URL:** `/academy/admin`

Доступна только пользователям с ролью **admin**. Позволяет управлять учебными материалами, создавать курсы и отслеживать прогресс студентов.

## 9. Безопасность

### 9.1 Хэширование паролей

Пароли пользователей хранятся в зашифрованном виде. Используется алгоритм bcrypt с **12 раундами соли**. Это обеспечивает высокую криптографическую стойкость при хранении.

> **Примечание для DevOps:** В целях оптимизации производительности на серверах с низким CPU рекомендуется снизить количество раундов до **10**. Изменение применяется через переменную окружения `BCRYPT_ROUNDS`.

### 9.2 Rate Limiting

Для предотвращения брутфорс-атак на эндпоинты аутентификации применяется ограничение частоты запросов:

- **Лимит:** 20 запросов с одного IP-адреса
- **Окно:** 15 минут

При достижении лимита сервер возвращает `429 Too Many Requests` с сообщением:  
`"Too many authentication attempts, please try again later."`

### 9.3 Заголовки безопасности

Приложение использует Helmet.js для установки защитных HTTP-заголовков. Content Security Policy настроена следующим образом:
- Разрешены скрипты только с доверенных CDN: tailwindcss.com, jsdelivr.net
- Разрешены изображения из `self`, `data:` и всех HTTPS-источников

### 9.4 JWT и срок жизни токена

JWT-токены выдаются при регистрации и входе. Срок жизни токена составляет **7 дней**. Для досрочного отзыва токена необходимо обратиться к администратору системы, так как механизм blacklist-токенов не реализован.

## 10. Коды ошибок и формат ответов

### 10.1 Стандартный формат ошибки

Все ошибки API возвращаются в едином формате:

```json
{
  "error": "Описание ошибки"
}
```

### 10.2 Таблица кодов HTTP-статусов

| Код | Описание | Когда возвращается |
|-----|----------|-------------------|
| 200 | OK | Успешный запрос |
| 400 | Bad Request | Невалидные данные в запросе |
| 401 | Unauthorized | Токен не передан |
| 403 | Forbidden | Токен недействителен или просрочен |
| 404 | Not Found | Ресурс не найден или не принадлежит пользователю |
| 429 | Too Many Requests | Превышен лимит запросов |
| 500 | Internal Server Error | Внутренняя ошибка сервера |

### 10.3 Примеры ошибочных ответов

**401 Unauthorized:**
```json
{
  "error": "Access token required"
}
```

**403 Forbidden:**
```json
{
  "error": "Invalid or expired token"
}
```

**404 Not Found (команда):**
```json
{
  "error": "Team not found"
}
```

## 12. Дополнительные правила и ограничения

### 12.1 Политика паролей

Система применяет единую политику паролей ко всем пользователям. Пароль должен соответствовать следующим требованиям:

- Минимальная длина — **8 символов**
- Наличие хотя бы одной заглавной буквы
- Наличие хотя бы одной цифры
- Запрещено использование пробелов

Данные требования применяются при регистрации через форму на сайте, а также при изменении пароля в личном кабинете. Серверная валидация возвращает ошибку `400 Bad Request` при несоответствии паролю установленным требованиям.

### 12.2 Жизненный цикл Secure Link

Защищённая ссылка для сотрудника имеет ограниченный срок действия. После генерации ссылка активна в течение **24 часов**. По истечении срока действия все запросы с использованием данного токена будут отклоняться с ошибкой `403 Forbidden`.

Менеджер может сгенерировать новую ссылку в любой момент. При генерации новой ссылки предыдущая немедленно аннулируется, даже если срок её действия ещё не истёк.

### 12.3 Ограничения команд и сотрудников

Система накладывает следующие ограничения на работу с командами:

- Максимальное количество команд на одного менеджера: **не ограничено**
- Максимальное количество сотрудников в одной команде: **10 человек**
- Сотрудник может состоять только в одной команде одновременно
- Менеджер видит только своих сотрудников и свои команды

При удалении сотрудника из команды (без удаления из системы) поле `teamId` у сотрудника устанавливается в `null`.

### 12.4 История навыков и права доступа

История изменений навыков сотрудника является конфиденциальной информацией. Политика доступа:

- **Менеджер** — полный доступ на чтение и запись через авторизованные эндпоинты
- **Сотрудник** — доступ только на чтение через авторизованные эндпоинты; просмотр через secure-токен **не поддерживается**
- **Анонимный пользователь** — доступ полностью закрыт

### 12.5 Логика определения типа DISC

После подсчёта баллов по четырём шкалам (D, I, S, C) система определяет тип личности следующим образом:

1. Если один тип явно доминирует (разница с ближайшим ≥ 4 баллов) — присваивается чистый тип (D, I, S или C)
2. Если разница между первым и вторым результатом ≤ 3 баллов — определяется комбинированный тип

Тест состоит из **15 вопросов** с четырьмя вариантами ответа в каждом. Каждый вариант ответа приносит **3 балла** в соответствующую шкалу. Максимально возможный балл по одной шкале — 45 (если на все вопросы выбраны ответы одного типа).
